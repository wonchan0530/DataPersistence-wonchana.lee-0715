# PRD.md

## 1. 개요

본 문서는 `DataPersistence` PoC 저장소의 제품 요구사항을 정의한다.

목표는 **JSON 파일을 이용해 데이터를 저장·불러오는 CRUD(Create, Read, Update, Delete) 콘솔 애플리케이션**을
개발하는 것이다. (`docs/[CRA_AI] Day2_1_Agentic Engineering.pdf` 29페이지, "[도전] PoC 만들기")

> 기존에 진행했던 PoC(Proof of Concept) 코드의 구조와 로직을 참고하여,
> 데이터를 JSON 파일로 관리하는 CRUD(Create, Read, Update, Delete) 콘솔 애플리케이션을 개발한다.
>
> - PoC에서 사용된 코드 구조를 유지한 상태로, CRUD 구현
> - Create: 새로운 데이터를 입력 받아 JSON 파일에 저장
> - Read: 전체 목록 보기 및 특정 ID/키 값으로 검색 기능
> - Update: 기존 데이터를 선택하여 특정 필드 수정
> - Delete: 특정 데이터를 안전하게 삭제

이 PoC는 이후 진행될 `SampleOrderSystem`(반도체 시료 생산주문관리 시스템) 프로젝트의 데이터 저장 계층으로
재사용되는 것을 전제로 하며, 여기서 검증된 JSON 영속성 구조를 그대로 가져다 쓸 수 있어야 한다.

## 2. 배경

- "데이터 영속성"이란 애플리케이션을 재시작해도 데이터가 유지되는 성질을 말한다.
  (`docs/[CRA_AI] Day3_개인과제_반도체시료관리.pdf` 25페이지)
- 팀/개인별로 선택한 방식(파일, JSON, DB 등)으로 데이터를 저장·불러오는 구조를 구현해야 하며, 본 과제에서는
  JSON 파일 방식을 채택한다.
- 관리 대상 데이터는 반도체 시료 생산주문관리 시스템의 핵심 데이터인 **시료(Sample)** 와 **주문(Order)** 을
  기반으로 한다. (`docs/[CRA_AI] Day3_개인과제_반도체시료관리.pdf` 12, 14페이지)

## 3. 사용자 및 사용 시나리오

콘솔 기반 애플리케이션으로, 담당자가 메뉴를 선택하고 값을 직접 입력하여 데이터를 조작한다.

- **시료 관리 담당자**: 신규 시료를 등록하고, 등록된 시료 목록/재고를 조회하거나 검색하고, 필요 시 시료 정보를
  수정·삭제한다.
- **주문 담당자**: 고객으로부터 받은 시료 요청을 주문으로 등록(예약)하고, 접수된 주문 목록을 조회·검색하며,
  필요 시 주문 내용을 수정하거나 취소(삭제)한다.

## 4. 데이터 모델 요구사항

### 4.1 시료 (Sample)

`docs/[CRA_AI] Day3_개인과제_반도체시료관리.pdf` 12페이지 "시료 관리" 기준.

| 필드 | 설명 | 비고 |
|---|---|---|
| 시료 ID | 시료를 식별하는 고유 값 | 필수, 유일값 |
| 이름 | 시료명 | 필수 |
| 평균 생산시간 | 시료 1개를 생산하는 데 걸리는 평균 시간(min/ea) | 필수, 숫자 |
| 수율 | 정상적인 시료 / 총 생산 시료 (예: 100개 생산 중 90개 정상 = 0.9) | 필수, 0~1 사이 숫자 |
| 현재 재고 | 현재 보유 중인 시료 수량 | 시료 조회 시 함께 표시, 최초 등록 시 0 또는 초기값 |

### 4.2 주문 (Order)

`docs/[CRA_AI] Day3_개인과제_반도체시료관리.pdf` 14페이지 "시료 주문" 기준.

| 필드 | 설명 | 비고 |
|---|---|---|
| 주문번호 | 주문을 식별하는 고유 값 | 필수, 유일값 |
| 시료 ID | 주문 대상 시료를 가리키는 참조 값 | 필수, 등록된 시료여야 함 |
| 고객명 | 주문한 고객 이름 | 필수 |
| 주문 수량 | 요청 수량 | 필수, 양의 정수 |
| 상태 | 주문 처리 상태 (예: RESERVED) | 예약 시점에는 RESERVED로 초기화 |

승인/거절, 생산, 출고 등 주문 상태 전이 로직(RESERVED → CONFIRMED/PRODUCING/RELEASED/REJECTED)은
`SampleOrderSystem` 프로젝트의 범위이며, 본 PoC에서는 CRUD 동작 검증을 위한 데이터 구조로만 다룬다.

## 5. 기능 요구사항

### 5.1 공통

- FR-1. 모든 데이터는 JSON 파일에 저장되어야 하며, 애플리케이션을 재시작해도 이전에 저장한 데이터를 그대로
  불러올 수 있어야 한다.
- FR-2. 시료 데이터와 주문 데이터는 각각 별도로 관리되어야 한다(별도 JSON 파일 또는 파일 내 별도 키 구조).
- FR-3. 콘솔에서 메뉴를 통해 Create/Read/Update/Delete 기능에 접근할 수 있어야 한다.
- FR-4. 잘못된 입력(존재하지 않는 ID 조회, 필수값 누락 등)에 대해 사용자에게 실패 사유를 안내해야 한다.

### 5.2 시료(Sample) CRUD

- FR-5. **Create**: 시료 ID, 이름, 평균 생산시간, 수율을 입력받아 새 시료를 등록하고 JSON 파일에 저장한다.
  이미 존재하는 시료 ID는 등록할 수 없다.
- FR-6. **Read (전체 조회)**: 등록된 모든 시료 목록을 현재 재고 수량과 함께 표시한다.
- FR-7. **Read (검색)**: 시료 ID 또는 이름 등 속성으로 특정 시료를 검색한다.
- FR-8. **Update**: 기존 시료를 선택하여 이름/평균 생산시간/수율/재고 등 특정 필드를 수정한다.
- FR-9. **Delete**: 특정 시료를 삭제한다. 삭제 전 존재 여부를 확인하고, 삭제 결과를 사용자에게 안내한다.

### 5.3 주문(Order) CRUD

- FR-10. **Create**: 시료 ID, 고객명, 주문 수량을 입력받아 새 주문을 생성(예약)하고 JSON 파일에 저장한다.
  존재하지 않는 시료 ID로는 주문을 생성할 수 없다.
- FR-11. **Read (전체 조회)**: 등록된 모든 주문 목록을 표시한다.
- FR-12. **Read (검색)**: 주문번호, 고객명, 시료 ID 등으로 특정 주문을 검색한다.
- FR-13. **Update**: 기존 주문을 선택하여 고객명/주문 수량/상태 등 특정 필드를 수정한다.
- FR-14. **Delete**: 특정 주문을 안전하게 삭제(취소)한다.

## 6. 비기능 요구사항

- NFR-1. 데이터 파일 입출력 중 오류(파일 없음, 파싱 실패 등)가 발생해도 애플리케이션이 비정상 종료되지 않고
  적절히 처리되어야 한다.
- NFR-2. 저장/조회 로직은 이후 `SampleOrderSystem`의 Model 계층에 재사용 가능하도록 독립된 모듈로 분리한다.
- NFR-3. 최초 실행 시 데이터 파일이 없는 경우 자동으로 빈 저장소 상태로 초기화되어야 한다.

## 7. 범위 외 (Out of Scope)

- 주문 승인/거절, 생산 라인 처리, 출고 처리 등 상태 전이 워크플로우 (→ `SampleOrderSystem`에서 다룸)
- 모니터링/통계 화면 (→ `DataMonitor`에서 다룸)
- 더미 데이터 자동 생성 (→ `DummyDataGenerator`에서 다룸)
- 동시성 제어, 다중 사용자 접근, 인증/권한
